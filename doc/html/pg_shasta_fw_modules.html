Firmware Modules

As introduced above, the Shasta FW architecture is heavily using the term "Firmware Module". A FW Module is an entity that implements a set of functionalities, using a well-defined set of input and output data and possibly keeping internal states and data. The characteristics of an Shasta FW module are defined in pgShasta_Coding and more specifically in sctShasta_Module_Design, so make sure you have read and understood these rules when working with FW modules.

  • System FW Module
    The SYS module provides the basic services like boot procedures, power management, inter-thread communication (if not covered by the RTOS itself) and other functionality that is considered "commonly" use throughout the rest of the FW.
    Its module prefix is sys_, ie. all its public functions are prefixed with osys_.
  • Patch FW Module
    The PATCH module provides the ability to apply patches to the ROM-based Shasta FW, ie. it allows to patch existing FW functionality without having to replace or re-spin the complete ROM content.
    Its module prefix is patch_, ie. all its public functions are prefixed with patch_.
  • RTX RTOS FW Module
    The RTOS is ARM's RTX. Its documentation is available in RTX User Manual.
    Its module prefix is os. Its prefix deviates from the sctShasta_Module_Design because it is a 3rd party off-the-shelf module from Keil/ARM.
  • PMBUS FW Module
    The PMBus module provides the handling of PMBUS commands that are received from master devices on the PMBus.
    Its module prefix is pmbus_, ie. all its functions are prefixed with pmbus_.
  • Regulation FW Module
    The regulation module provides functions for real-time control of voltage and current.
  • Telemetry FW Module
    The telemetry module provides functions for get the status of the conroller i.e. voltage and current information.
  • Faults FW Module
    The telemetry module provides functions handle and detect fault in the conroller i.e. over_voltage, under_current.
  • Logger FW Module
    The Logging module takes care of logging all information (including errors, warnings and status messages) within the FW that is relevant for the outside.
    Its module prefix is log_, ie. all its public functions are prefixed with log_.
  • pgShasta_job_agency
    The job agency provides a generic environment for inter-module communication.
  • Configurator FW Module
    This digital DCDC loop DC/DC configuration module manages the DCDC module configuration sets. These configuration sets are applied during system startup. The firmware module provides PMBus messages to modify, add and apply configuration sets. It also supports select the configuration which need to be applied during startup.
  • Operation, Administration & Maintenance Module
    The Operation and Maintenance (OAM) module does the FW-wide house-keeping in the sense that it aggregates and processes data from the other modules. Furthermore, it contains the top-level or rather device-level exception handling and decides what to do when other modules report error conditions.
    Its module prefix is oam_, ie. all its public functions are prefixed with oam_.
  • DMA Driver module
    The DMA driver layer provides access routines to direct memory access module provided by ARM.
  • DTIMER Driver module
    The dtimer driver layer provides access routines to the two independent dual timer HW modules provided by ARM. Each module consists of 2 timer channels which provide periodic and single shot functionality.
  • MMU Driver module
    The memory management unit (mmu) driver provides access to the HW MMU module. It allows to re-map memory regions in 1kByte blocks
  • OTP Driver Module
    The OTP driver provides access to the OTP. OTP is a one-time programmable hardware in Shasta. OTP is 64 KBs size with up to seventeen partitions. The first partition holds trim and configuration data, the remaining partitions hold FW patches.
  • UART FW Module
    The telemetry module provides functions handle and detect fault in the conroller i.e. over_voltage, under_current.
